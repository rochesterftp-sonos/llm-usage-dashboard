Overview

This dashboard provides real-time monitoring of LLM API usage for Level 2 Compliance Solutions. Security and data protection are critical priorities for this application.

API Key Security

  • Server-Side Only: API keys are never transmitted to your browser or exposed in frontend code
  • Environment Variables: Keys are stored as encrypted environment variables on Railway's secure platform
  • No Client Access: Frontend code only receives aggregated usage data, never credentials
  • HTTPS Encryption: All communication between your browser and the server is encrypted via TLS

Authentication

  • Password Protection: Dashboard access requires password authentication
  • Session Management: Secure session tokens with 24-hour expiration
  • No Public Access: Unauthorized users cannot view usage data or make API requests

Data Handling

  • No PII Storage: This dashboard does not collect, store, or transmit personally identifiable information
  • Aggregated Metrics Only: Only token counts and costs are trackedâ€”no message content, prompts, or responses
  • In-Memory Cache: Usage data is stored temporarily in server memory (last 100 data points)
  • No Database: No persistent storage of sensitive data; all data resets on server restart

âš ï¸ Potential Threats & Mitigations

Threat: Unauthorized Dashboard Access

Risk: Someone guesses or obtains the dashboard password

Mitigation:

  • Use a strong, unique password (set in DASHBOARD_PASSWORD env var)
  • Change password regularly
  • Dashboard URL is not publicly listed or indexed
  • Monitor Railway logs for unusual access patterns

Impact if Compromised: Attacker can see usage metrics but cannot access API keys or modify data

Threat: Railway Platform Breach

Risk: Compromise of Railway hosting infrastructure

Mitigation:

  • Railway uses industry-standard security practices (SOC 2 Type II compliance)
  • Environment variables encrypted at rest
  • Regular security audits by Railway team

Impact if Compromised: Potential exposure of API keys; rotate keys immediately if breach detected

Threat: Man-in-the-Middle (MITM) Attack

Risk: Interception of data between browser and server

Mitigation:

  • TLS/HTTPS encryption for all traffic
  • Railway automatically provisions SSL certificates
  • Modern cipher suites and protocols enforced

Impact if Compromised: Attacker could see usage data and session token; cannot see API keys (server-side only)

Threat: API Key Leakage via Logs

Risk: API keys accidentally logged to console or error messages

Mitigation:

  • Server code does not log API keys
  • Error messages sanitized to remove sensitive data
  • Railway logs are access-controlled

Impact if Compromised: Unauthorized API usage; rotate affected keys immediately

ğŸ” Best Practices

  • Use a Strong Password: Minimum 16 characters, mix of letters/numbers/symbols
  • Don't Share the URL: Keep the dashboard link private
  • Monitor Usage: Regularly check for unexpected spikes in token usage
  • Rotate API Keys: Periodically rotate API keys (quarterly recommended)
  • Log Out: Always log out when done, especially on shared devices
  • Enable 2FA: Enable two-factor authentication on OpenAI, Anthropic, and OpenRouter accounts

ğŸ“ Incident Response

If you suspect a security breach:

  1. Immediately change the DASHBOARD_PASSWORD in Railway environment variables
  2. Rotate all affected API keys (OpenAI, Anthropic, OpenRouter)
  3. Review Railway access logs for unusual activity
  4. Check provider dashboards for unexpected API usage
  5. Document the incident and timeline
